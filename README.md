## Постановка задачи 
Проведите анализ разнородных геоданных о зданиях и предложите подход к построению единой модели высотности, пригодной для задач планирования
сети. Для этого:
  1. Проанализируйте качество данных, предоставленных из источников, и предложите методы очистки.
  2. Разработайте способ сопоставления объектов, относящихся к одному физическому зданию, в условиях, когда один дом в разных источниках может
  быть представлен по-разному.
  3. Предложите алгоритм или модель определения наиболее достоверной высоты здания и обоснуйте свой выбор.
     
   Если ни один из источников не содержит информации о высоте здания, предложите алгоритм ее оценки по косвенным признакам (например, на
    основе площади застройки, назначения здания, этажности соседних зданий, данных геокодера или иных открытых данных). Обоснуйте примени-
    мость предложенного подхода.

  5. Проведите валидацию полученных результатов.

  6. Визуализируйте итоговый слой зданий на карте и сформулируйте инсайты о структуре застройки города, которые могут быть полезны для планирова-
ния сети.

Назначение модуля

Данный модуль представляет собой законченный пайплайн предобработки геоданных, разработанный для решения задачи предсказания высотности зданий на основе разнородных источников информации. Модуль принимает на вход два независимых источника данных в формате CSV, где геометрия зданий представлена в текстовом формате WKT (Well-Known Text), и производит последовательную обработку: загрузку, валидацию, очистку, пространственное сопоставление, объединение атрибутов и генерацию признаков для последующего обучения моделей машинного обучения.

## Ключевые задачи, решаемые модулем
Модуль решает комплекс взаимосвязанных задач, каждая из которых критически важна для получения качественных данных:
  -  Задача нормализации разнородных источников. Два источника данных могут использовать разные схемы именования колонок, разные кодировки текста, разные способы представления геометрии и разные системы координат. Модуль приводит все данные к единому формату, обеспечивая согласованность дальнейшей обработки.
  -  Задача обеспечения качества геометрии. Реальные геоданные практически всегда содержат ошибки: самопересекающиеся полигоны, разрывы в контурах, вырожденные геометрии, пустые значения. Модуль реализует многоуровневую систему валидации и автоматического исправления геометрических ошибок, основанную на анализе типичных проблем в открытых геоданных.
  -  Задача семантической фильтрации. Не все объекты в исходных данных являются целевыми для анализа. Гаражи, сараи, теплицы, временные постройки часто присутствуют в кадастровых выгрузках, но не представляют интереса для задачи предсказания высотности капитальных зданий. Модуль реализует гибкую систему фильтрации на основе тегов, атрибутов и эвристических правил.
  -  Задача пространственного сопоставления. Один и тот же физический объект может присутствовать в обоих источниках с разными идентификаторами и атрибутами. Модуль реализует двухэтапный алгоритм матчинга: сначала попытка сопоставления по пересечению полигонов с расчётом коэффициента IoU, затем — сопоставление по близости центроидов с использованием эффективной структуры данных cKDTree для ускорения поиска.
  -  Задача разрешения конфликтов при слиянии. При объединении данных из двух источников возникают ситуации, когда одному объекту из источника А соответствуют несколько кандидатов из источника Б. Модуль реализует стратегию разрешения many-to-one конфликтов на основе максимизации коэффициента пространственного соответствия.
  -  Задача обогащения данных признаками. Для эффективного обучения моделей машинного обучения сырых атрибутов недостаточно. Модуль генерирует широкий спектр признаков: геометрические характеристики формы здания, пространственные признаки положения в городе, статистики окружения на разных радиусах, кодированные категориальные признаки, статистические нормализации.
## Архитектурные принципы
  -  Принцип единственной ответственности. Каждый модуль в пакете решает строго определённую задачу. Конфигурация отвечает только за хранение настроек, загрузчик — только за чтение и парсинг файлов, очиститель — только за валидацию и фильтрацию, мерджер — только за сопоставление и слияние, инженер признаков — только за генерацию фичей, главный скрипт — только за оркестрацию. Это позволяет тестировать, отлаживать и модифицировать каждый компонент независимо.
  -  Принцип явной конфигурации. Все параметры, влияющие на поведение пайплайна, вынесены в централизованный конфигурационный модуль. Это позволяет настраивать пороги фильтрации, списки тегов, радиусы поиска и другие параметры без изменения кода бизнес-логики. Конфигурация использует типизированные классы данных, что обеспечивает проверку типов на этапе разработки и автодополнение в средах разработки.
  -  Принцип прозрачности и наблюдаемости. Каждый значимый этап обработки сопровождается детальным логированием. Модули собирают статистику о количестве обработанных записей, проценте отфильтрованных объектов, времени выполнения операций. Это позволяет диагностировать проблемы, настраивать пороги и понимать, как изменения в конфигурации влияют на результат.
  -  Принцип устойчивости к ошибкам. Пайплайн спроектирован так, чтобы продолжать работу при возникновении некорректных данных в отдельных записях. Ошибки парсинга, невалидные геометрии, пропущенные значения обрабатываются грациозно: проблемная запись исключается из обработки с записью в лог, но не приводит к остановке всего процесса.
  -  Принцип воспроизводимости. Все случайные процессы (если таковые появятся в будущих расширениях) должны использовать фиксированные seed-значения. Все преобразования данных детерминированы при фиксированной конфигурации. Результаты выполнения пайплайна могут быть воспроизведены на другом окружении при наличии тех же исходных данных и конфигурации.
## Модуль конфигурации (config.py)
### Назначение и общая структура
Модуль конфигурации является фундаментом всего пайплайна. Он определяет все параметры, управляющие поведением остальных компонентов, и обеспечивает централизованное управление настройками. Конфигурация реализована с использованием механизма dataclasses из стандартной библиотеки Python, что обеспечивает типобезопасность, автодополнение в редакторах кода и удобство документирования.
Структура конфигурации иерархическая: корневой класс Config содержит ссылки на вложенные конфигурационные классы CleaningConfig и FeatureConfig, а также определяет глобальные настройки проекта, такие как пути к файлам и параметры логирования.

Класс CleaningConfig: параметры очистки данных

Класс CleaningConfig инкапсулирует все параметры, управляющие процессом предобработки и фильтрации геоданных.
  -  Параметр MIN_AREA_SQM определяет минимальную допустимую площадь здания в квадратных метрах. Объекты с площадью меньше этого порога считаются артефактами оцифровки или нецелевыми объектами (например, маленькими сооружениями) и исключаются из дальнейшего анализа. Значение по умолчанию установлено в 10 квадратных метров, что соответствует минимальному разумному размеру капитального строения.
  -  Параметр MAX_AREA_SQM определяет максимальную допустимую площадь здания. Этот порог защищает от ошибок в данных, когда площадь может быть указана с ошибкой на несколько порядков (например, из-за неправильной единицы измерения или опечатки). Значение 50000 квадратных метров выбрано как разумный верхний предел для отдельных зданий в городской застройке.
  -  Параметры MIN_HEIGHT_M и MAX_HEIGHT_M аналогично ограничивают допустимый диапазон высоты зданий. Минимальная высота в 2 метра отсекает объекты, которые не могут быть зданиями (заборы, маленькие будки), а максимальная высота в 200 метров соответствует высоте самых высоких жилых и офисных зданий в большинстве городов России.
  -  Параметр OUTLIER_ZSCORE_THRESHOLD управляет статистической фильтрацией выбросов. Метод основан на расчёте Z-score — стандартизированного отклонения значения от среднего, выраженного в количествах стандартных отклонений. Значение порога 3.0 означает, что отбрасываются только те значения, которые отклоняются от среднего более чем на три стандартных отклонения, что соответствует классическому правилу трёх сигм для нормального распределения.
   -  Параметр SHED_TAGS представляет собой список строковых ключевых слов, используемых для идентификации вспомогательных построек. Список включает русскоязычные и англоязычные варианты названий: гараж, сарай, теплица, навес, а также их английские эквиваленты. Этот список используется для семантической фильтрации: если в тегах или описании назначения здания встречается любое из этих слов, объект помечается как нецелевой и исключается из анализа.
  -  Параметр MIN_IOU_THRESHOLD определяет минимальный коэффициент пересечения (Intersection over Union), необходимый для признания двух геометрических объектов соответствующими друг другу. IoU рассчитывается как отношение площади пересечения двух полигонов к площади их объединения. Порог 0.1 означает, что даже небольшое пространственное перекрытие считается достаточным для потенциального сопоставления, что важно при работе с данными разной точности оцифровки.
  -  Параметр MAX_DISTANCE_METERS задаёт максимальное расстояние между центроидами двух объектов, при котором они ещё могут рассматриваться как кандидаты на сопоставление. Значение 50 метров выбрано как компромисс: достаточно большое, чтобы учесть погрешности геокодирования и различия в системах координат, но достаточно малое, чтобы избежать ложных сопоставлений в плотной городской застройке.
  -  Параметр MAX_AREA_RATIO ограничивает допустимое отношение площадей двух сопоставляемых объектов. Если площадь одного объекта превышает площадь другого более чем в пять раз, они с большой вероятностью представляют разные физические сущности, даже если их геометрии пересекаются. Этот параметр предотвращает ошибочное сопоставление, например, маленького гаража и большого жилого дома, расположенных рядом.
  -  Параметры CRS_INPUT и CRS_METRIC определяют системы координат, используемые на разных этапах обработки. CRS_INPUT со значением EPSG:4326 указывает, что входные данные используют географическую систему координат WGS84 (широта и долгота в градусах). CRS_METRIC со значением EPSG:3857 указывает на использование проекции Web Mercator для всех метрических расчётов (площадь, расстояние, периметр), так как в градусах эти величины не имеют физического смысла.

Класс FeatureConfig: параметры генерации признаков

Класс FeatureConfig управляет процессом извлечения и конструирования признаков для моделей машинного обучения.
  -  Параметр GEOMETRY_FEATURES содержит список имён геометрических признаков, которые будут вычислены для каждого объекта. В список включены: площадь в квадратных метрах, периметр, компактность (отношение площади к квадрату периметра, нормированное на коэффициент формы круга), вытянутость (соотношение сторон ограничивающего прямоугольника), количество вершин полигона, площадь выпуклой оболочки и отношение площади объекта к площади его выпуклой оболочки. Каждый из этих признаков несёт информацию о форме и размере здания, что может быть полезно для классификации типа строения или предсказания его характеристик.
  -  Параметр NEIGHBOR_RADII_METERS определяет набор радиусов для анализа окружения каждого здания. Для каждого радиуса из списка (25, 50, 100, 200 метров) вычисляются статистики соседних объектов: количество соседей, средняя высота, средняя площадь, плотность застройки. Такой многоуровневый анализ позволяет модели учитывать контекст: здание в плотной исторической застройке может иметь иные характеристики, чем аналогичное по форме здание в пригороде.
  -  Параметр BUILDING_TYPES представляет собой словарь, отображающий категории типов зданий на списки ключевых слов для их распознавания в текстовых полях исходных данных. Ключами словаря являются канонические названия категорий (residential, office, retail, public, industrial, shed), а значениями — списки слов и морфем, которые могут встречаться в описаниях зданий на русском и английском языках. Этот словарь используется для классификации зданий по назначению на основе семантического анализа текстовых полей.
  -  Параметр STATISTICAL_FEATURES определяет список статистических преобразований, применяемых к числовым признакам. Включены Z-score (стандартизация) и квантильное ранжирование для площади и высоты. Эти преобразования приводят признаки к сопоставимым масштабам, что важно для многих алгоритмов машинного обучения, чувствительных к различиям в диапазонах значений.
  -  Параметр SPATIAL_FEATURES содержит список пространственных признаков, описывающих положение здания в городе. Включены расстояние до центра города, нормированные координаты и индекс геохеша системы H3. Эти признаки позволяют модели учитывать географический контекст: здания в центре города могут иметь иные характеристики, чем здания на окраинах.

Класс Config: основная конфигурация проекта

Корневой класс Config объединяет все настройки и управляет инфраструктурными аспектами проекта.
  -  Параметр PROJECT_ROOT автоматически определяет корневую директорию проекта как родительскую директорию относительно файла конфигурации. Это делает проект переносимым: его можно клонировать в любое место файловой системы, и все относительные пути будут разрешаться корректно без необходимости ручной настройки.
  -  Параметры DATA_DIR, RESULTS_DIR, LOGS_DIR определяют пути к директориям для исходных данных, результатов работы и файлов логов соответственно. Все пути строятся относительно PROJECT_ROOT, обеспечивая согласованную структуру проекта.
  -  Параметры SOURCE_A и SOURCE_B указывают конкретные файлы исходных данных для двух источников. Эти пути используются загрузчиком для чтения CSV-файлов.
  -  Параметры CLEANING и FEATURES являются экземплярами соответствующих конфигурационных классов. Они предоставляют удобный доступ к настройкам очистки и генерации признаков через атрибуты, например, config.CLEANING.MIN_AREA_SQM или config.FEATURES.NEIGHBOR_RADII_METERS.
  -  Параметры LOG_LEVEL и LOG_FORMAT управляют детализацией и форматом записей в журнале событий. Уровень логирования может быть изменён для отладки (DEBUG) или для продакшена (INFO, WARNING), а формат определяет, какая информация включается в каждую запись лога.
  -  Метод post_init является специальным методом dataclass, который автоматически вызывается после создания экземпляра класса. В этом методе реализуется автоматическое создание директорий результатов и логов, если они ещё не существуют. Это обеспечивает работоспособность пайплайна «из коробки» без необходимости предварительной ручной подготовки структуры папок.

Глобальный экземпляр конфигурации

В конце модуля создаётся глобальный экземпляр конфигурации: config = Config(). Этот экземпляр импортируется во всех остальных модулях пайплайна, обеспечивая единый источник истины для всех настроек. Использование глобального экземпляра оправдано тем, что конфигурация по своей природе является общим состоянием приложения, и создание множественных экземпляров с разными настройками в рамках одного запуска пайплайна не предусмотрено.
# Модуль загрузки данных (data_loader.py)
Назначение модуля

Модуль data_loader.py отвечает за чтение исходных данных из файлов, парсинг геометрических представлений и преобразование данных в формат GeoDataFrame библиотеки GeoPandas. Модуль предоставляет как низкоуровневые утилитарные функции для безопасного парсинга, так и высокоуровневые функции для загрузки целых файлов с комплексной обработкой ошибок.

Функция safe_wkt_loads: безопасный парсинг WKT-строк

Функция safe_wkt_loads принимает на вход строку в формате WKT (Well-Known Text) и возвращает геометрический объект библиотеки Shapely или None в случае ошибки.
  -  Обработка отсутствующих значений. Функция первым делом проверяет, является ли входное значение пустым: pd.isna(wkt_str) для обработки пропущенных значений pandas, сравнение с пустой строкой и с None для обработки явных пустых значений. Если значение отсутствует, функция немедленно возвращает None, предотвращая попытки парсинга некорректных данных.
  -   Нормализация входной строки. Входное значение преобразуется в строку и очищается от ведущих и завершающих пробелов методом strip(). Затем проверяется, не является ли строка одним из текстовых представлений отсутствующего значения: 'none', 'nan', 'null' в любом регистре. Это необходимо, так как в CSV-файлах пропущенные значения могут быть записаны как текст.
  -   Парсинг WKT. Основная операция парсинга выполняется функцией wkt.loads() из библиотеки Shapely. Эта функция преобразует текстовое представление геометрии в объект соответствующего класса (Polygon, MultiPolygon, Point и так далее).
  -   Валидация геометрии. После успешного парсинга геометрия проверяется на пустоту методом is_empty. Пустые геометрии, такие как GEOMETRYCOLLECTION EMPTY, не несут полезной информации и отбрасываются.
  -   Автоматическое исправление невалидных геометрий. Если геометрия не проходит проверку is_valid, что может указывать на самопересечения или другие топологические ошибки, применяется функция make_valid() для попытки автоматического исправления. Если исправление не удаётся или результат оказывается пустым, функция возвращает None.
  -   Фильтрация по типу геометрии. Модуль предназначен для работы с площадными объектами (зданиями), поэтому принимаются только геометрии типов Polygon и MultiPolygon. Точки, линии и другие типы отбрасываются, так как они не могут представлять здания в контексте данной задачи.
  -   Обработка исключений. Вся логика парсинга обернута в блок try/except, который перехватывает любые исключения и возвращает None. Это гарантирует, что ошибка в одной записи не приведёт к остановке обработки всего файла. Детали ошибки записываются в лог на уровне DEBUG для последующего анализа при отладке.
Функция load_source: загрузка файла в GeoDataFrame

Функция load_source является основным интерфейсом модуля для загрузки данных. Она принимает путь к файлу и имя колонки с геометрией, а возвращает объект GeoDataFrame, готовый к дальнейшей обработке.
  -  Проверка существования файла. Функция преобразует входной путь в объект Path и проверяет существование файла. Если файл не найден, выбрасывается исключение FileNotFoundError с информативным сообщением. Это позволяет быстро диагностировать проблемы с путями к данным.
  -  Загрузка CSV с автоопределением кодировки. Функция пытается загрузить файл в кодировке UTF-8, которая является стандартом для современных данных. Если возникает ошибка UnicodeDecodeError, что характерно для файлов в кодировке Windows-1251 (cp1251), часто используемой в российских системах, функция автоматически повторяет загрузку с указанием кодировки cp1251. Такой подход избавляет пользователя от необходимости вручную определять кодировку файла.
  -  Поиск колонки с геометрией. Функция ищет колонку, содержащую геометрические данные, перебирая список возможных имён: геометрия, wkt, WKT, geom, geometry_wkt. Это обеспечивает совместимость с разными схемами данных, где колонка геометрии может называться по-разному. Если ни одна из ожидаемых колонок не найдена, выбрасывается исключение ValueError со списком доступных колонок для помощи в отладке.
  -  Применение безопасного парсинга. Для каждой строки в найденной колонке геометрии применяется функция safe_wkt_loads. Результат записывается в новую колонку geometry. Это создаёт серию геометрических объектов, готовых для использования в GeoDataFrame.
  -  Фильтрация валидных геометрий. Создаётся булева маска, отмечающая строки, для которых парсинг геометрии завершился успешно (значение не None). Подсчитывается количество валидных геометрий и выводится в лог процент успешных записей относительно общего количества строк. Если валидных геометрий не осталось вообще, выбрасывается исключение, так как дальнейшая обработка невозможна.
  -  Создание GeoDataFrame. Функция создаёт объект GeoDataFrame библиотеки GeoPandas, указывая колонку geometry как геометрическую и устанавливая систему координат из конфигурации (CRS_INPUT, то есть EPSG:4326). При создании удаляется исходная текстовая колонка с геометрией, так как она больше не нужна после парсинга.
  -  Логирование результата. Функция записывает в лог итоговое количество объектов в созданном GeoDataFrame и используемую систему координат. Это позволяет отследить, сколько данных было успешно загружено после всех этапов фильтрации.

Функция add_basic_features: добавление базовых признаков

Функция add_basic_features добавляет в GeoDataFrame набор базовых вычисляемых признаков, которые потребуются на последующих этапах обработки.
  -  Создание копии данных. Функция начинает с создания копии входного GeoDataFrame, чтобы избежать побочных эффектов изменения исходных данных.
  -  Проекция для метрических расчётов. Для вычисления площади и других метрических характеристик геометрия временно проецируется в метрическую систему координат (CRS_METRIC, то есть EPSG:3857). Это необходимо, так как в географических координатах (градусах) площадь и расстояние не имеют физического смысла.
  -  Вычисление центроидов. Функция вычисляет координаты центроида (геометрического центра) каждого полигона в метрической проекции и сохраняет их в колонках centroid_x и centroid_y. Также сохраняется объект центроида как геометрия в колонке centroid для удобства последующих операций.
  -  Вычисление площади. Площадь каждого полигона вычисляется в метрической проекции и сохраняется в колонке area_calculated_sqm. Это значение используется для фильтрации, сопоставления и генерации признаков.
  -  Логарифмическое преобразование площади. Создаётся признак log_area, представляющий собой натуральный логарифм от площади плюс единица (функция np.log1p). Логарифмирование сглаживает распределение площадей, которое обычно имеет длинный хвост в сторону больших значений, что делает признак более пригодным для использования в моделях машинного обучения.
  -  Добавление метки источника. В датафрейм добавляется колонка data_source, содержащая переданное название источника. Это позволяет в дальнейшем различать происхождение записей после слияния данных из нескольких источников.
  -  Логирование. Функция записывает в лог сообщение о завершении добавления признаков для указанного источника.
  
# Модуль очистки данных (data_cleaner.py)
Назначение и архитектура класса DataCleaner

Класс DataCleaner реализует комплексный пайплайн очистки геоданных. Он инкапсулирует всю логику валидации, исправления и фильтрации записей, предоставляя простой интерфейс для загрузки и очистки файла в один вызов. Класс спроектирован с возможностью обработки нескольких источников независимо: каждый экземпляр класса ведёт собственную статистику и логирует операции с указанием имени источника.

Метод init: инициализация очистителя

Метод инициализации принимает строковый параметр source_name, который используется для идентификации источника в логах и статистике. Также метод инициализирует словарь stats, который будет накапливать статистику по количеству записей на каждом этапе очистки: initial (исходное количество), after_geometry_clean (после исправления геометрий), after_shed_filter (после фильтрации вспомогательных построек), after_outlier_filter (после удаления статистических выбросов), final (итоговое количество). Такая детализация позволяет анализировать, на каком этапе теряется больше всего данных, и при необходимости корректировать пороги фильтрации.
Метод load_and_clean: главный публичный интерфейс
Метод load_and_clean является точкой входа для пользователя класса. Он принимает путь к файлу и имя колонки с геометрией, а возвращает очищенный GeoDataFrame.
Последовательность этапов. Метод последовательно вызывает приватные методы для каждого этапа очистки: _load_data для загрузки и парсинга, _clean_geometries для исправления геометрических ошибок, _filter_sheds для удаления вспомогательных построек, _filter_outliers для статистической фильтрации. После каждого этапа обновляется соответствующее поле в словаре статистики.
Логирование и статистика. По завершении всех этапов вызывается метод _log_stats, который выводит в лог сводную статистику очистки: исходное количество записей, количество после каждого этапа, итоговый процент сохранённых записей. Это обеспечивает прозрачность процесса и помогает диагностировать проблемы с данными.
Возврат результата. Метод возвращает очищенный GeoDataFrame, готовый к дальнейшей обработке или экспорту.
Метод _load_data: загрузка данных с обработкой ошибок
Метод реализует логику, вынесенную из модуля data_loader.py, но адаптированную для использования внутри класса. Он загружает CSV-файл, определяет колонку с геометрией, применяет безопасный парсинг WKT и создаёт GeoDataFrame с указанной системой координат. Ключевое отличие от функции load_source — использование конфигурации через атрибут config.CLEANING.CRS_INPUT и интеграция со статистикой экземпляра класса.
Метод _safe_wkt_loads: внутренний парсер геометрий
Метод является копией функции safe_wkt_loads из data_loader.py, но реализован как метод класса для удобства вызова через self. Логика полностью идентична: обработка отсутствующих значений, нормализация строки, парсинг WKT, валидация, автоматическое исправление, фильтрация по типу геометрии, обработка исключений.
Метод _clean_geometries: исправление семи типов геометрических ошибок
Метод реализует комплексную очистку геометрий, направленную на устранение семи типичных проблем, перечисленных в техническом задании.
Внутренняя функция clean_geom. Для каждой геометрии применяется вложенная функция, которая последовательно проверяет и исправляет возможные проблемы.
Обработка пустых геометрий. Если геометрия отсутствует или помечена как пустая, функция возвращает None, что приведёт к исключению записи на следующем этапе.
Исправление самопересечений. Если геометрия не проходит проверку is_valid, применяется функция make_valid() для автоматического исправления топологических ошибок. Если исправление не удаётся или результат пуст, возвращается None.
Обработка MultiPolygon. Если геометрия имеет тип MultiPolygon (состоит из нескольких полигонов), из всех компонентов выбирается полигон с максимальной площадью. Это эвристическое правило основано на предположении, что для задачи анализа зданий основной интерес представляет крупнейший компонент, а мелкие фрагменты могут быть артефактами оцифровки.
Фильтрация по минимальной площади. Геометрия временно проецируется в метрическую систему координат, и вычисляется её площадь. Если площадь меньше 1 квадратного метра, геометрия отбрасывается. Этот порог выбран как минимальный разумный размер для капитального строения.
Возврат результата. Если геометрия прошла все проверки, она возвращается для включения в результат.
Применение к датафрейму. Функция clean_geom применяется ко всем геометрия в датафрейме через метод apply. Затем датафрейм фильтруется, оставляя только строки с не-пустыми геометрия. Подсчитывается и логируется количество удалённых записей.
Метод _filter_sheds: фильтрация вспомогательных построек
Метод реализует многоуровневую стратегию идентификации и удаления нецелевых объектов: гаражей, сараев, теплиц и других малых построек.
Внутренняя функция is_shed. Для каждой строки датафрейма применяется функция, возвращающая True, если объект должен быть отфильтрован.
Проверка по тегам источника А. Функция извлекает значение колонки tags, преобразует его в строку нижнего регистра и проверяет наличие любого из ключевых слов из списка config.CLEANING.SHED_TAGS. Если совпадение найдено, объект помечается как вспомогательная постройка.
Проверка по назначению источника Б. Аналогичная проверка выполняется для колонки purpose_of_building, которая может присутствовать во втором источнике данных.
Эвристическая проверка по площади и этажности. Если объект имеет ровно один этаж (stairs или gkh_floor_count_max равно 1) и площадь меньше 30 квадратных метров, он помечается как вероятная вспомогательная постройка. Этот критерий основан на статистическом анализе: большинство гаражей и сараев в российской застройке имеют именно такие параметры, тогда как даже небольшие жилые дома обычно начинаются от 35-40 квадратных метров.
Применение маски фильтрации. Функция is_shed применяется ко всем строкам датафрейма, создавая булеву маску. Подсчитывается количество отфильтрованных объектов и выводится в лог процент относительно количества записей до этого этапа. Датафрейм фильтруется, оставляя только объекты, не помеченные как вспомогательные постройки.
Метод _filter_outliers: удаление статистических выбросов
Метод реализует фильтрацию записей с экстремальными значениями площади и высоты на основе статистических критериев.
Фильтрация по площади. Если в датафрейме присутствует колонка с площадью (area_sq_m или area_calculated_sqm), метод вычисляет квантили распределения: первый квартиль (25-й процентиль) и третий квартиль (75-й процентиль). На основе этих значений вычисляется межквартильный размах (IQR = Q3 - Q1). Границы допустимого диапазона определяются как [Q1 - 3IQR, Q3 + 3IQR]. Коэффициент 3 выбран как более мягкий по сравнению с классическим 1.5, чтобы не отфильтровать реальные крупные объекты (торговые центры, промышленные здания). Дополнительно границы ограничиваются значениями из конфигурации (MIN_AREA_SQM и MAX_AREA_SQM), чтобы исключить явно нереалистичные значения даже при необычном распределении данных. Создаётся булева маска, отмечающая записи в допустимом диапазоне, и датафрейм фильтруется.
Фильтрация по высоте. Если присутствует колонка height, применяется аналогичный алгоритм на основе квантилей и IQR. Важное отличие: записи с отсутствующим значением высоты (NaN) не отбрасываются, а сохраняются, так как отсутствие данных не является ошибкой. Это реализуется через логическое ИЛИ с условием isna() при создании маски фильтрации.
Логирование результата. Подсчитывается количество удалённых записей и выводится в лог процент потерь.
Метод _log_stats: вывод статистики очистки
Метод выводит в лог сводную таблицу результатов очистки для данного источника. Для каждого этапа указывается количество записей, а для итогового значения дополнительно вычисляется процент сохранённых записей относительно исходного количества. Форматирование с разделителями тысяч (через :,) улучшает читаемость больших чисел.
Метод export_cleaned: экспорт результатов
Метод сохраняет очищенный датафрейм в двух форматах для максимальной совместимости.
Экспорт в GeoJSON. Метод to_file с драйвером GeoJSON сохраняет данные в формате, поддерживающем геометрию и атрибуты, и совместимом с большинством ГИС-инструментов (QGIS, ArcGIS, веб-карты).
Экспорт в CSV с WKT. Для совместимости с инструментами, не работающими с гео-форматами (Excel, pandas, системы аналитики), геометрия конвертируется в текстовый формат WKT и сохраняется в отдельной колонке. Используется кодировка utf-8-sig с BOM для корректного открытия кириллических данных в Excel на Windows.
Логирование. Путь к сохранённому файлу записывается в лог для отслеживания результатов.
Модуль слияния данных (data_merge.py)
Назначение и архитектура класса DataMerger
Класс DataMerger отвечает за пространственное сопоставление объектов из двух источников, разрешение конфликтов при множественных соответствиях, слияние атрибутов и запуск генерации признаков. Класс инкапсулирует сложную логику матчинга и предоставляет простой интерфейс merge для выполнения всего процесса в один вызов.
Метод init: инициализация мерджера
Метод инициализирует атрибуты для хранения промежуточных результатов: matches_df для датафрейма с найденными соответствиями, merged_gdf для итогового объединённого датафрейма. Также создаётся экземпляр класса FeatureEngineer для генерации признаков и словарь stats для накопления статистики.
Метод merge: главный публичный интерфейс
Метод принимает два очищенных GeoDataFrame (источники А и Б) и возвращает объединённый датафрейм с признаками.
Логирование входных данных. Метод записывает в лог количество объектов в каждом источнике для отслеживания масштаба данных.
Пространственное сопоставление. Вызывается метод _spatial_match, который находит пары объектов из разных источников, представляющих одно и то же здание. Результат сохраняется в атрибут matches_df.
Разрешение many-to-one конфликтов. Вызывается метод _resolve_many_to_one, который для каждого объекта из источника А выбирает наилучшее соответствие из источника Б на основе максимального коэффициента IoU. Это необходимо, так как из-за различий в оцифровке одному полигону могут соответствовать несколько кандидатов.
Слияние атрибутов. Вызывается метод _merge_attributes, который объединяет атрибуты из обоих источников в один датафрейм, добавляя префикс к колонкам из источника Б для избежания конфликтов имён.
Логика MAX для высоты. Вызывается метод _apply_max_height_logic, который вычисляет итоговую высоту здания как максимум из доступных источников: прямое значение высоты из источника Б, высота, рассчитанная как этажность на среднюю высоту этажа, высота из источника А. Это позволяет получить наиболее полную оценку высоты при наличии частичных данных в разных источниках.
Генерация признаков. Вызывается метод extract_all_features экземпляра FeatureEngineer, который добавляет в датафрейм все вычисляемые признаки для машинного обучения.
Логирование статистики. Вызывается метод _log_stats, который выводит сводную информацию о результатах слияния.
Возврат результата. Метод возвращает итоговый GeoDataFrame, готовый для обучения модели или экспорта.
Метод _spatial_match: двухэтапное пространственное сопоставление
Метод реализует гибридный алгоритм поиска соответствий, комбинирующий точное сопоставление по пересечению и приближённое сопоставление по центроидам.
Проекция в метрическую систему. Оба входных датафрейма проецируются в метрическую систему координат для корректного расчёта расстояний и площадей.
Этап 1: сопоставление по пересечению. Используется функция gpd.sjoin с предикатом intersects для нахождения всех пар объектов, геометрии которых пересекаются. Для каждой найденной пары вычисляется коэффициент IoU и расстояние между центроидами. Если IoU превышает порог из конфигурации, пара добавляется в список совпадений с меткой типа match_type = 'intersection'.
Обработка ошибок sjoin. Операция sjoin обернута в try/except, так как на больших данных или при сложных геометриях могут возникать ошибки. В случае ошибки метод логирует предупреждение и продолжает работу со вторым этапом.
Этап 2: сопоставление по центроидам для оставшихся объектов. Из источника А отбираются объекты, для которых не было найдено соответствий на первом этапе. Для них вызывается метод _centroid_match, который ищет ближайшие объекты в источнике Б по центроидам с использованием эффективной структуры данных cKDTree.
Возврат результата. Найденные совпадения преобразуются в датафрейм с колонками id_a, id_b, iou, distance_m, match_type. Если совпадений не найдено, возвращается пустой датафрейм с заданной структурой колонок.
Метод _calculate_iou: расчёт коэффициента пересечения
Метод вычисляет Intersection over Union для двух геометрий: площадь пересечения делится на площадь объединения. Для избежания деления на ноль при очень малых площадях добавляется проверка: если объединение меньше 1e-6, возвращается 0.0. Вся логика обернута в try/except для обработки возможных геометрических ошибок.
Метод _centroid_match: сопоставление по центроидам с cKDTree
Метод реализует эффективный поиск ближайших соседей для оставшихся без соответствий объектов.
Построение KD-дерева. Координаты центроидов всех объектов источника Б преобразуются в массив и используются для построения структуры cKDTree, которая обеспечивает логарифмическое время поиска ближайшего соседа.
Поиск ближайших. Для каждого центроида объекта из источника А выполняется запрос к дереву для нахождения ближайшего объекта в источнике Б. Возвращаются расстояние и индекс найденного объекта.
Фильтрация по расстоянию. Если расстояние не превышает порог из конфигурации, пара считается потенциальным соответствием. Дополнительно вычисляется IoU для оценки качества соответствия.
Возврат списка совпадений. Метод возвращает список словарей, каждый из которых описывает найденное соответствие с метаданными.
Метод _resolve_many_to_one: разрешение конфликтов множественных соответствий
Метод решает проблему, когда одному объекту из источника А соответствуют несколько объектов из источника Б.
Стратегия выбора. Для каждого уникального id_a выбирается соответствие с максимальным значением IoU. Это реализуется через группировку датафрейма по id_a и применение idxmax() к колонке iou.
Логирование результата. Метод выводит в лог количество уникальных идентификаторов в каждом источнике после разрешения конфликтов, что позволяет оценить, насколько успешно прошло сопоставление.
Метод _merge_attributes: слияние атрибутов с префиксами
Метод объединяет атрибуты из двух источников в единый датафрейм.
Копирование источника А. За основу берётся датафрейм источника А, в который добавляется метка data_source = 'A'.
Создание маппинга соответствий. Датафрейм matches преобразуется в словарь, где ключом является id_a, а значением — словарь с id_b, iou и match_type. Это позволяет быстро получать информацию о соответствии для любого объекта.
Добавление колонок соответствия. В датафрейм добавляются колонки matched_id_b, match_iou, match_type, которые для каждого объекта из источника А содержат информацию о найденном соответствии в источнике Б или None, если соответствие не найдено.
Слияние атрибутов источника Б. Для каждой колонки в источнике Б, которая не является геометрией и не дублирует колонки источника А, создаётся новая колонка с префиксом B_. Значения заполняются путём поиска по id в индексированном датафрейме источника Б.
Определение источника высоты. Добавляется колонка height_source, которая указывает, из какого источника взято значение высоты: из источника Б, если оно присутствует, или вычислено из источника А.
Логирование. Метод выводит в лог количество добавленных атрибутов из источника Б.
Метод _apply_max_height_logic: вычисление итоговой высоты
Метод реализует эвристическую логику выбора наилучшей оценки высоты здания из доступных источников.
Функция calculate_max_height. Для каждой строки собираются все доступные оценки высоты: прямое значение из колонки B_height, расчётная высота как произведение этажности (B_stairs) на среднюю высоту этажа (B_avg_floor_height или дефолтное значение 3.0), расчётная высота из источника А как произведение gkh_floor_count_max на 3.0. Из всех доступных оценок выбирается максимальное значение. Если оценок нет, возвращается None.
Применение к датафрейму. Функция применяется ко всем строкам через метод apply, результат сохраняется в колонку final_height_m.
Статистика. Подсчитывается количество зданий с определённой высотой и количество случаев, когда для выбора высоты использовалось сравнение нескольких оценок (функция max). Результаты выводятся в лог.
Метод _log_stats: вывод статистики слияния
Метод выводит сводную информацию: количество найденных соответствий, количество уникальных идентификаторов в каждом источнике, итоговый размер объединённого датафрейма, количество сгенерированных признаков.
Метод export_merged: экспорт результатов
Метод сохраняет итоговый датафрейм в форматах GeoJSON и CSV с WKT, аналогично методу export_cleaned класса DataCleaner. Дополнительно вызывается метод export_feature_stats для сохранения статистики по признакам.
Модуль генерации признаков (feature_engineering.py)
Назначение и архитектура класса FeatureEngineer
Класс FeatureEngineer отвечает за конструирование признаков из сырых атрибутов и геометрий для использования в моделях машинного обучения. Класс реализует модульную архитектуру: каждый тип признаков генерируется отдельным приватным методом, что обеспечивает читаемость, тестируемость и лёгкость расширения.
Метод init: инициализация инженера признаков
Метод инициализирует словарь label_encoders для хранения обученных кодировщиков категориальных признаков, словарь stats для статистики и атрибуты для кэширования структуры соседства (neighbors_tree, neighbors_data).
Метод extract_all_features: главный публичный интерфейс
Метод принимает очищенный и объединённый GeoDataFrame и возвращает его же с добавленными признаками.
Последовательность генерации. Метод последовательно вызывает приватные методы для каждого типа признаков: _add_geometry_features, _add_spatial_features, _add_neighborhood_features, _add_categorical_features, _add_statistical_features, _add_address_features. Порядок выбран так, чтобы признаки, зависящие от других (например, статистические, требующие уже вычисленной площади), генерировались после базовых.
Условная генерация признаков окружения. Признаки, требующие анализа соседних объектов, генерируются только если предоставлен полный датафрейм gdf_all. Если он не передан, метод логирует предупреждение и пропускает этот этап.
Сбор статистики. После генерации всех признаков метод собирает список колонок, начинающихся с префикса feat_, и сохраняет их количество и имена в словаре stats.
Возврат результата. Метод возвращает обогащённый датафрейм.
Метод _add_geometry_features: геометрические признаки
Метод вычисляет признаки, описывающие форму и размер здания.
Проекция для метрических расчётов. Геометрия проецируется в метрическую систему для корректного вычисления площади и периметра.
Логарифм площади. Признак feat_log_area вычисляется как логарифм площади плюс единица для сглаживания распределения.
Периметр. Признак feat_perimeter содержит длину границы полигона.
Компактность. Признак feat_compactness вычисляется по формуле 4πS/P², где S — площадь, P — периметр. Для идеального круга значение равно 1, для вытянутых форм — меньше. Добавление малого значения 1e-6 в знаменатель предотвращает деление на ноль.
Вытянутость. Признак feat_elongation вычисляется как отношение сторон ограничивающего прямоугольника. Функция max(x, 1/x) гарантирует, что значение всегда не меньше 1, независимо от ориентации здания.
Количество вершин. Признак feat_vertices_count подсчитывает количество точек в контуре полигона. Для мультиполигонов возвращается 0, так как обработка усложнена.
Площадь выпуклой оболочки. Признак feat_convex_hull_area содержит площадь минимального выпуклого полигона, содержащего исходную геометрию.
Отношение площади к выпуклой оболочке. Признак feat_convex_hull_ratio показывает, насколько форма здания близка к выпуклой. Значение 1 означает идеально выпуклую форму.
Метод _add_spatial_features: пространственные признаки
Метод добавляет признаки, описывающие положение здания в городе.
Определение центра города. Для Санкт-Петербурга центр задаётся как координаты Дворцовой площади. Координаты преобразуются в метрическую проекцию для расчёта расстояний.
Расстояние до центра. Признак feat_distance_to_center_km содержит расстояние от центроида здания до центра города в километрах.
Координаты центроида. Признаки feat_x_coord и feat_y_coord содержат координаты центроида в метрической проекции.
H3-индекс. Признак feat_h3_index содержит индекс геохеша системы H3 с разрешением 9. Для вычисления используется библиотека h3, если она установлена. Если библиотека отсутствует, метод логирует предупреждение и присваивает значение 0.
Метод _add_neighborhood_features: признаки окружения
Метод вычисляет статистики соседних объектов на нескольких радиусах.
Построение KD-дерева. Для всех объектов в датасете строится cKDTree по координатам центроидов для эффективного поиска соседей.
Определение доступных колонок. Метод проверяет наличие колонок с высотой и площадью в исходном датафрейме и определяет их фактические имена.
Цикл по радиусам. Для каждого радиуса из конфигурации:
Для каждого целевого объекта находится список индексов соседей в заданном радиусе через query_ball_point.
Подсчитывается количество соседей (минус сам объект).
Вычисляется средняя высота соседей, игнорируя пропущенные значения.
Вычисляется средняя площадь соседей.
Вычисляется плотность застройки как количество соседей на гектар площади круга заданного радиуса.
Результаты сохраняются в колонки с суффиксом, указывающим радиус.
Метод _add_categorical_features: категориальные признаки
Метод кодирует текстовые категории в числовые признаки.
Извлечение типа здания. Если присутствуют колонки tags или purpose_of_building, метод _extract_building_type применяется к их значениям для определения канонического типа здания на основе словаря из конфигурации.
Label Encoding типа здания. Если режим обучения (is_train=True), создаётся и обучается LabelEncoder, который сохраняется в словаре label_encoders. Если режим инференса, используется сохранённый энкодер; неизвестные категории получают код -1.
Признак жилой/нежилой. Метод _is_residential проверяет наличие ключевых слов жилого назначения в тегах или назначении и создаёт бинарный признак feat_is_residential.
Кодирование района. Если присутствует колонка district, она кодируется аналогично типу здания. Если колонки нет, признак получает значение 0.
Метод _add_statistical_features: статистические признаки
Метод вычисляет стандартизированные версии числовых признаков.
Z-score и квантиль площади. Если присутствует колонка площади, вычисляются среднее и стандартное отклонение. Z-score вычисляется как (значение - среднее) / отклонение. Квантиль вычисляется как процентный ранг значения в распределении. Если данных недостаточно (менее 10 записей), признакам присваиваются дефолтные значения.
Аналогично для высоты. Если присутствует колонка высоты, вычисляются те же статистики.
Метод _add_address_features: адресные признаки
Метод создаёт признаки, описывающие полноту адресной информации.
Наличие адреса. Признак feat_has_address равен 1, если колонка gkh_address не пуста.
Наличие номера дома. Признак feat_has_number равен 1, если колонка number не пуста.
Полнота адреса. Признак feat_address_completeness вычисляется как среднее двух предыдущих бинарных признаков.
Вспомогательные методы
Метод _extract_building_type перебирает категории из конфигурации и возвращает первую, ключевые слова которой найдены во входном тексте. Если совпадений нет, возвращается 'other'.
Метод _is_residential проверяет наличие ключевых слов жилого типа в тегах или назначении строки.
Метод get_feature_columns возвращает список имён сгенерированных признаков.
Метод export_feature_stats вычисляет и сохраняет базовую статистику по каждому признаку: среднее, стандартное отклонение, минимум, максимум, количество и долю пропусков.
Точка входа (main.py)
Назначение модуля
Модуль main.py является точкой входа в пайплайн. Он отвечает за инициализацию логирования, последовательный запуск всех этапов обработки и вывод итоговой статистики.
Функция setup_logging: настройка логирования
Функция настраивает корневой логгер проекта: устанавливает уровень логирования из конфигурации, формат записей, добавляет обработчики для записи в файл и вывода в консоль. Файл лога открывается в режиме перезаписи (mode='w'), чтобы каждый запуск начинался с чистого лога.
Функция main: оркестрация пайплайна
Функция принимает аргументы командной строки и выполняет последовательность шагов.
Шаг 1: Очистка источника А. Создаётся экземпляр DataCleaner для источника А, вызывается метод load_and_clean для загрузки и очистки файла SOURCE_A, результат экспортируется в results/cleaned_source_A.csv.
Шаг 2: Очистка источника Б. Аналогично для источника Б, с указанием geometry_col='wkt', так как во втором источнике колонка геометрии может называться иначе.
Шаг 3: Объединение и генерация признаков. Создаётся экземпляр DataMerger, вызывается метод merge для сопоставления, слияния и обогащения данных, результат экспортируется в CSV и GeoJSON.
Вывод итоговой статистики. Функция выводит в лог сводную информацию: исходное и итоговое количество записей для каждого источника, количество найденных соответствий, размер объединённого датасета, долю записей с высотой, количество сгенерированных признаков, пути к файлам результатов.
Возврат результатов. Функция возвращает словарь с промежуточными датафреймами и статистикой для возможного использования в интерактивной среде.
Блок if name == "main"
Обрабатывает аргументы командной строки через argparse, вызывает функцию main, проверяет результат и завершает процесс с соответствующим кодом возврата.

Структура результатов
После успешного выполнения в директории results появятся файлы:
cleaned_source_A.csv — очищенные данные источника А
cleaned_source_B.csv — очищенные данные источника Б
merged_dataset.csv — объединённый датасет с признаками (геометрия в WKT)
merged_dataset.geojson — объединённый датасет в формате GeoJSON
feature_stats.csv — статистика по сгенерированным признакам
pipeline.log — журнал выполнения
Зависимости и требования
Основные зависимости
pandas>=1.5.0: работа с табличными данными, загрузка CSV, манипуляции с датафреймами
geopandas>=0.12.0: геопространственные операции, проекции, пространственные соединения
shapely>=2.0.0: геометрические примитивы, операции с полигонами, валидация и исправление геометрий
numpy>=1.24.0: математические операции, работа с массивами
scipy>=1.9.0: структура данных cKDTree для эффективного поиска соседей
scikit-learn>=1.2.0: LabelEncoder для кодирования категориальных признаков
h3>=3.7.0 (опционально): генерация H3-индексов для пространственной кластеризации
Системные требования
Оперативная память: рекомендуется не менее 8 ГБ для обработки датасетов до 100 000 объектов
Дисковое пространство: зависит от размера исходных данных, результаты занимают примерно 2-3x от размера исходных файлов
Процессор: многоядерный процессор ускорит выполнение, хотя пайплайн не использует явную параллелизацию
